Visão Geral do Sistema

O EcoTrack é uma plataforma SaaS B2B voltada para restaurantes, clínicas e pequenas indústrias que precisam gerir o descarte de resíduos em conformidade com a Política Nacional de Resíduos Sólidos (PNRS — Lei 12.305/2010) e a Resolução CONAMA 275/2001. A plataforma conecta três pilares da cadeia de resíduos:

Perfil Papel no Sistema Principais Necessidades
Gerador
Restaurantes, Clínicas		 Origem do resíduo — solicita coletas e emite MTR Conformidade legal, relatórios de sustentabilidade, facilidade no descarte
Transportador
Coletores certificados		 Logística de coleta — aceita rotas e assina MTR Otimização de rotas, gestão de frota, documentação digital
Destinador
Aterros, Recicladoras		 Recebimento final — confirma e emite CDF Controle de entrada, emissão de Certificado de Destinação Final
🎯
Proposta de Valor Automatizar 100% do ciclo documental de resíduos (MTR + CDF) eliminando papel, reduzindo risco de autuação ambiental e provendo evidências para ESG corporativo.

Perfis de Usuário e Roles

Cada empresa cadastrada na plataforma possui usuários com roles que determinam o nível de acesso:

RolePermissõesRestrições
ADMIN Acesso total: gerenciar empresa, usuários, plano, exclusão de conta (LGPD) Não pode rebaixar a própria role; não pode desativar a si mesmo
OPERADOR Agendar coletas, emitir MTR, visualizar relatórios e inventário Não pode gerenciar usuários nem alterar dados da empresa
MOTORISTA Visualizar rotas atribuídas, atualizar status da coleta para EM_ROTA e COLETADO Não pode criar coletas, acessar relatórios ou configurações
  • RN-USR-01 Um ADMIN não pode alterar sua própria role para OPERADOR ou MOTORISTA, evitando que a empresa fique sem administrador.
  • RN-USR-02 Um ADMIN não pode desativar seu próprio usuário via endpoint DELETE /me/usuarios/:id.
  • RN-USR-03 Ao desativar um usuário, todos os seus refresh tokens são revogados imediatamente (cascade no banco).
  • RN-USR-04 Email de usuário deve ser único em toda a plataforma (não apenas dentro da empresa).

  • RN-MTR-01 Todo agendamento de coleta cria automaticamente um ManifestoMTR com status RASCUNHO vinculado à coleta (relação 1:1 via coletaId).
  • RN-MTR-02 Um MTR só pode ser emitido (RASCUNHO → EMITIDO) se a coleta estiver nos status PENDENTE, CONFIRMADA ou EM_ROTA.
  • RN-MTR-03 Não é possível emitir um novo MTR se a coleta já possui um MTR com status diferente de RASCUNHO (evita duplicatas).
  • RN-MTR-04 Fase 1: número MTR gerado localmente no formato MTR-{ANO}-{6 últimos chars do coletaId}. Fase 2: número oficial via API SINIR.
  • RN-MTR-05 O MTR aceita 3 assinaturas digitais independentes: GERADOR, TRANSPORTADOR e DESTINADOR. Cada assinatura é armazenada em campo separado.
  • RN-MTR-06 Quando as 3 assinaturas estiverem presentes, o status é automaticamente atualizado para FINALIZADO e o campo finalizadoEm é preenchido.
  • RN-MTR-07 O endpoint GET /:id/pdf retorna dados estruturados conforme CONAMA 275/2001 para geração de PDF pelo frontend, incluindo metadado de conformidade.

Regras de Negócio — Autenticação

5.1 Tokens JWT

  • RN-AUTH-01 Access token JWT válido por 1 hora. Payload contém: sub (usuarioId), empresaId, role, exp.
  • RN-AUTH-02 Refresh token opaco de 7 dias armazenado no banco de dados (tabela refresh_tokens). Um por sessão ativa por usuário.
  • RN-AUTH-03 Rotação de refresh token: a cada uso do /auth/refresh, o token antigo é revogado e um novo é emitido. Reutilização de token revogado é recusada com HTTP 401.
  • RN-AUTH-04 Senhas armazenadas com bcrypt (custo 12). Nunca armazenadas em texto puro.
  • RN-AUTH-05 Login retorna HTTP 401 com mensagem genérica ("Credenciais inválidas") tanto para email inexistente quanto para senha errada — não revela qual campo está incorreto.
  • RN-AUTH-06 Logout requer envio do refreshToken no body para revogação imediata no banco. Access token expira naturalmente.
  • RN-AUTH-07 Usuários com ativo: false não conseguem fazer login — retorna HTTP 401 com código USER_INATIVO.

5.2 Cadastro de Empresa

  • RN-CAD-01 CNPJ deve ser único na plataforma. Tentativa de cadastro com CNPJ já existente retorna HTTP 409 com código CNPJ_JA_CADASTRADO.
  • RN-CAD-02 Email do usuário admin deve ser único em toda a plataforma. Retorna HTTP 409 com código EMAIL_JA_CADASTRADO.
  • RN-CAD-03 Cadastro cria atomicamente: Empresa + Usuário ADMIN + RefreshToken em uma única transação Prisma. Em caso de falha, nada é persistido.
  • RN-CAD-04 Novo cadastro retorna automaticamente access e refresh tokens — o usuário já fica logado ao se cadastrar (sem etapa adicional de login).

Regras de Negócio — Coletas

6.1 Criação de Coleta

  • RN-COL-08 Todos os resíduos informados na criação devem pertencer ao inventário ativo da empresa geradora. Resíduo não cadastrado no inventário retorna HTTP 400 com código RESIDUO_NAO_ENCONTRADO.
  • RN-COL-09 O endereço de coleta é opcional. Se não informado, o sistema usa o endereço cadastrado na empresa. O endereço é copiado (snapshot) no momento do agendamento — alterações posteriores na empresa não afetam coletas existentes.
  • RN-COL-10 A criação da coleta gera atomicamente: registro Coleta + ColetaResiduo[] + ManifestoMTR (RASCUNHO) + ColetaStatusHistorico (entrada inicial). Falha em qualquer etapa reverte toda a transação.
  • RN-COL-11 Data/hora de agendamento deve estar no futuro e no formato ISO 8601. Coletas no passado são rejeitadas na validação de schema.

6.2 Listagem e Paginação

  • RN-COL-12 Listagem aceita filtros: status, dataInicio, dataFim. Paginação padrão: page=1, limit=20. Limite máximo: 100 itens por página.
  • RN-COL-13 Toda listagem retorna meta com: page, limit, total, totalPages.

Regras de Negócio — Inventário de Resíduos

  • RN-INV-01 O catálogo de resíduos é público (sem autenticação). Baseado na classificação ABNT NBR 10.004 e lista IBAMA. Campos: tipo, descrição, código IBAMA, classe ABNT, periculosidade.
  • RN-INV-02 Cada empresa possui seu próprio inventário (vínculo InventarioResiduo). Um mesmo resíduo não pode ser adicionado duas vezes ao inventário da mesma empresa (unique constraint empresaId + residuoId).
  • RN-INV-03 Se o resíduo já estiver no inventário mas inativo (ativo: false), a operação de adição o reativa ao invés de criar um novo registro, atualizando quantidade, unidade e frequência.
  • RN-INV-04 Remoção do inventário é soft delete (ativo: false). O histórico de coletas vinculado ao resíduo é preservado.
  • RN-INV-05 Tipos de resíduos suportados: ORGANICO, RECICLAVEL, ELETRONICO, HOSPITALAR, PERIGOSO. Unidades de medida: KG, LITRO, UNIDADE.
  • RN-INV-06 Frequências de geração: DIARIA, SEMANAL, QUINZENAL, MENSAL, SOB_DEMANDA.

Regras de Negócio — MTR (Manifesto de Transporte de Resíduos)

📋
Base Legal O Manifesto de Transporte de Resíduos (MTR) é exigido pela Resolução CONAMA 275/2001 e deve conter dados completos do gerador, transportador, tipo de resíduo, volume e destinação final.
  • RN-MTR-08 Multi-tenancy no MTR é garantido via JOIN com a tabela coletas: a query filtra por coleta.empresaId = empresaId do JWT, impedindo acesso a manifestos de outras empresas.
  • RN-MTR-09 Assinatura requer que o MTR esteja em status EMITIDO ou ACEITO. Tentativa de assinar em RASCUNHO ou FINALIZADO retorna HTTP 422.
  • RN-MTR-10 O endpoint de PDF retorna JSON estruturado com: manifesto (número, status, datas), gerador (CNPJ, endereço completo), transportador (licença ambiental), coleta.residuos, e metadado de conformidade CONAMA 275/2001.
  • RN-MTR-11 Todo acesso ao PDF do MTR gera AuditLog com ação DOWNLOAD_PDF_MTR (LGPD — controle de acesso a dados pessoais do gerador e transportador).

Regras de Negócio — Gestão de Empresa

9.1 Atualização de Dados

  • RN-EMP-01 Apenas usuários com role ADMIN podem atualizar dados da empresa (PATCH /empresas/me). Campos imutáveis via API: CNPJ, tipo, plano, razão social principal.
  • RN-EMP-02 Requisição PATCH com body vazio retorna HTTP 400 com código EMPTY_BODY — evita operações nulas no banco.

9.2 Exclusão de Conta (LGPD Art. 18)

⚖️
Conformidade LGPD — Direito à Exclusão O endpoint DELETE /empresas/me/delete implementa o direito de exclusão previsto no Art. 18 da LGPD. A exclusão é irreversível e anonimiza todos os dados pessoais.
  • RN-EMP-03 A exclusão de conta executa em transação atômica: (1) revogar todos os refresh tokens; (2) desativar todos os usuários; (3) anonimizar empresa (email, CNPJ, razão social); (4) anonimizar emails de usuários; (5) criar AuditLog de exclusão.
  • RN-EMP-04 Dados anonimizados: email → deletado_{timestamp}@anonimizado.ecotrack; CNPJ → 00.000.000/0000-XX; razão social → [CONTA ENCERRADA {timestamp}].
  • RN-EMP-05 Registros de MTR são mantidos anonimizados para conformidade CONAMA — obrigação legal de manter rastro de destinação de resíduos. Não são excluídos.
  • RN-EMP-06 O AuditLog de exclusão (ação EXCLUSAO_CONTA_LGPD) é mantido mesmo após a anonimização para rastreabilidade de auditoria regulatória.

9.3 Gestão de Usuários

  • RN-EMP-07 Criação de usuário via API requer: email único, nome (mín. 2 chars), senha (mín. 8 chars), role. Senha é hasheada com bcrypt antes de persistir.
  • RN-EMP-08 Listagem de usuários retorna apenas usuários com ativo: true — usuários desativados são ocultados.

Multi-tenancy e Segurança

  • RN-SEC-01 Multi-tenancy: Toda query ao banco filtra por empresaId extraído do JWT. É impossível para um usuário acessar dados de outra empresa — mesmo com um ID válido de recurso alheio.
  • RN-SEC-02 MTRs são buscados via JOIN na tabela coletas (coleta.empresaId = empresaId), garantindo isolamento mesmo sem campo direto de empresa no manifesto.
  • RN-SEC-03 Todas as rotas (exceto GET /residuos catálogo público) exigem JWT válido via hook onRequest: [fastify.authenticate].
  • RN-SEC-04 Rotas administrativas (PATCH empresa, DELETE empresa, CRUD usuários) exigem hook adicional fastify.authenticateAdmin que valida role === "ADMIN".
  • RN-SEC-05 Padrão de resposta de erro unificado: { success: false, error: { code: "CODIGO", message: "Descrição" } }. Códigos de erro nunca expõem detalhes internos.
  • RN-SEC-06 Padrão de resposta de sucesso: { success: true, data: {}, meta: {} }. Meta contém paginação quando aplicável.
🔒
Isolamento de Dados O EcoTrack implementa Row-Level Security via aplicação (não via PostgreSQL RLS nativo). O empresaId extraído do JWT assinado é a fonte de verdade para todo isolamento — ele não pode ser sobrescrito por parâmetros da requisição.
